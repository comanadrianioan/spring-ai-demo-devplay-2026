What are AI agents?

In simple terms, an AI agent is a system that acts on its own to achieve a goal. Unlike a chatbot that only answers once, an agent can plan tasks, make decisions, and take actions over time. Think of giving the AI a mission—it figures out the steps and can adjust along the way. Instead of only asking what the weather is, you might ask it to plan a weekend trip, suggest hotels, and remind you to pack—and it tries to carry that through.

Agent = Model + Harness

The software wrapped around the model — managing its environment, context, and external tool access so it can actually act.

The AI engineering role has shifted three times in four years, and each shift moved the optimization target. In twenty-twenty-two and twenty-twenty-three, the question was: how do I ask the model? That was prompt engineering — roles, examples, reasoning patterns. A small prompt change could dramatically improve results, so we optimized instructions. By twenty-twenty-four, teams realized most failures came from missing context, not bad prompts. The question became: what information should the model have? RAG, memory, conversation history, tool outputs — we optimized information. Now, in twenty-twenty-five and twenty-twenty-six, the model itself is just one component. The question is: how do I organize the entire system around the model? Agent orchestration, tool calling, memory management, evaluation loops. We optimize the execution environment. As models improved, the bottleneck moved — from prompt quality, to context quality, to system design. That is the harness. And an AI Agent is simply a Model plus a Harness. Watch for that pattern in every architecture iteration we walk through — each one is really a more capable harness around the same LLM.

Software architectures with AI

Why

More data than rules-only systems can handle.

How

LLMs in the stack—intent in, useful output out.

Agents are getting smart enough for production, and expectations are rising. Why now? Because the amount of data we process is huge—traditional software cannot keep up, while AI can learn and adapt. How do we use it? By integrating large language models into our applications so they understand what users mean and help produce clear, useful responses quickly.

Retrieval‑Augmented Generation

To face those challenges we often use RAG—retrieval augmented generation. RAG connects the LLM to a library of knowledge. When someone asks a question, we search that library first, then feed the relevant facts to the model together with the question. That way the answer is grounded in data you control. This diagram is the prompt cycle: retrieve, augment the prompt, then generate—not sending the question to the model in isolation.

RAG benefits

More relevant, factual outputs
Update knowledge without full retraining
Lower cost when designed with guardrails
Grounding the model on real data cuts down hallucinations. You can refresh the knowledge base without retraining the entire model. Done right, with proper guards against abuse, it can also be cheaper. Now that we have the basics of LLMs and RAG, Razvan will take you through a real-world scenario we keep running into.

MCP — Model Context Protocol

A quick aside before the architectures. Without MCP, every agent has to speak each platform's own API — Slack one way, Drive another, Git a third — and you end up maintaining N times M custom connectors. With MCP, the agent speaks one protocol; each tool ships an MCP server that does the translation. The blast radius of a new tool drops from "build a custom integration" to "spin up an MCP server". Keep this in mind when we get to the agentic orchestration diagram later — that is where MCP earns its keep.

Real-world scenario

Picture this: you hear a Teams notification and already feel the context switch coming. In a feature channel, someone asks how to configure rewards distribution—again. Either you are in flow on something new and do not want to stop, or it has been so long that you would need to dig through wiki pages. You are a developer and an AI enthusiast, so you wonder whether you can automate the answer—and you heard agents are the trend for twenty-twenty-six.

Naive solution

So we sketch the simplest possible design. A chatbot registered to the Teams channel forwards the question through a gateway into our Chatbot App. We have plenty of wiki pages from past work, so we fetch all that data and serve it to the LLM alongside the question. This is an overly simplified example — in reality wiki data would also go through a gateway and loading it into memory on every request is a bad idea — but bear with us. We discover that we only need a good-enough prompt: all the wiki data plus the question from the channel, send it to the LLM, and forward the response back to the chatbot. Quick prototype that works — for now.

Benefits
Challenges
Processing a large amount of data in one shot

What we achieved: we process a lot of data at once, so the chatbot does not have to first locate the right wiki page. But adding all that data risks exceeding the LLM's maximum context window. The simple solution gives us a fast time to market — at the cost of forwarding every single question to the LLM, so the bill explodes the moment two people start chatting in the channel. We use prompt engineering to keep answers focused, yet with so much unrelated wiki content in the context we still see hallucinations and wrong answers. Walk through each benefit and its matching challenge as you advance.

RAG with semantic caching

Back to the drawing board. We remember what Adrian said about RAG, and we decide to implement semantic caching. We take all the wiki pages, run them through a document vectorization step — there are plenty of third-party apps for this — and store the result in a vector database. Now when a question arrives, we do a semantic similarity search to fetch only the chunks that match, use them as focused context for the LLM, and return the response. Again, an oversimplification — in reality we would need triggers to keep the DB in sync with wiki updates — but the architecture is much sharper than the naive version.

Benefits & challenges

Benefits
Challenges
Enhancing prompts with specific, focused context
Relying on a 3rd-party vectorization solution
Fast specific data retrieval from the vector DB
Additional data storage costs
Reduces the frequency of model retraining
More complex architecture
Let us see how we improved. We now enhance prompts with focused, question-specific data — but to create the vectors we depend on a third-party solution that can be expensive or inaccurate. We retrieve only wiki data related to the question, fast, thanks to the database — though we have introduced a new component: external storage. By feeding only relevant data to the LLM, we lean less on its general knowledge and reduce hallucinations — yet the architecture is more complex and the per-request LLM cost has not improved yet. So we go back to the drawing board for one more iteration.

RAG with semantic cache & chat history

Now we layer chat history on top of the semantic search from the previous iteration. The goal is still to cut LLM cost. When a question arrives, we first search a vector database of past question-and-answer pairs. If the similarity is above our threshold, we return the cached answer and skip the LLM entirely. Only on a miss do we fall back to the wiki RAG path: vector-search the wiki, build a rich prompt with the retrieved context, call the LLM, then vectorize and store the new Q+A pair in chat history before responding. The Chatbot App in this diagram orchestrates both the cache lookup and the retrieval path — we pay the LLM only when no past answer fits.

Benefits & challenges

Benefits
Challenges
Reduced cost of LLM requests — our main goal
Must fine-tune the similarity threshold or return partial answers
Faster responses when the cache hits — a DB search beats an LLM round-trip
Outdated responses when wiki data changes — needs a sync mechanism
Fault tolerance — history can answer when the LLM is unavailable
Overall complexity increased again
We achieved what we set out to do: cut the cost of LLM calls and speed up responses, because the fastest answer now comes straight from the database. We also bought ourselves a layer of fault tolerance — if the LLM is down, the chat-history DB can still answer. The trade-offs: we have to fine-tune the similarity threshold carefully to avoid partial answers; if the wiki changes in the meantime, cached responses go stale and we need a sync mechanism to invalidate them; and the overall architecture is more complex again. Agentic orchestration

We moved from a POC-style pipeline toward something more production-shaped. The important distinction: there is one orchestrator agent inside the Chatbot Orchestrator-it decides which tools to call and merges results. RAG search, chat history lookup, and web search are tools, not separate agents. The agent routes the request, invokes tools, builds the best prompt, calls the LLM when needed, and returns a confident answer.

Benefits & challenges

Benefits
Challenges
Solves the partial / outdated answer problem of the previous iteration
Longer response time — data travels through more components
Modular — extend or swap one tool without touching the rest
Needs sophisticated infrastructure for CD and high availability
Extra fault tolerance — web search can fall back when other tools cannot answer
Architecture has become even more complex
We successfully addressed the main drawback of the previous iteration — partial responses and outdated data — by routing smarter and enriching context. The cost is response time: data has to travel through multiple components. The big modularity win is that we can add or extend a single tool without redeploying or impacting the rest of the system — but that demands an equally sophisticated infrastructure for continuous deployment and high availability. We also picked up additional fault tolerance via the web search tool: when other tools cannot answer, it kicks in. The trade-off is obvious from the diagram — the architecture is even more complex.
